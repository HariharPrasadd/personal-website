Horus Architecture

Horus is an open-source, plug-and-play computer use agent (CUA). It can turn any LLM with vision and language capabilities into a computer use agent.

1. Core Design Principles

  • Plug-and-play - swap any component (grounding model, backbone VLM, OS control layer) without touching the rest
  • Local-first - perception runs locally, reasoning can be local or API
  • Benchmark-driven - every architectural addition must move OSWorld score, otherwise it's cut
  • License-clean - AGPL components isolated behind process boundaries, Horus core targets MIT/Apache 2.0
  • No multi-rollout - single rollout with recovery, not best-of-N (incompatible with real deployment)

Why not Electron?

Tauri uses the OS-native WebView + Rust backend. Result: ~10MB app vs ~200MB Electron bundle.

Rust also gives native OS API access for input injection without Node.js addon complexity.

Why Python for the brain?

The entire ML ecosystem (OmniParser, fastembed, smolagents, anthropic SDK) is Python.

Fighting this is not worth it. Rust handles the parts Python is bad at (OS calls, latency-sensitive input).

4. Perception Layer

What OmniParser actually is

Two models, not one:

Component Architecture Size License Role
icon_detectYOLOv8 Nano fine-tuned on UI~100MBAGPLDetects all interactive regions
icon_captionFlorence-2 Base~230MBMITLabels each detected region semantically

Total: ~330MB. Runs locally. Not heavy.

What it returns

A flat list - no tree, no hierarchy:

[
  { "id": 1, "bbox": [120, 45, 200, 65], "label": "File menu" },
  { "id": 2, "bbox": [1380, 870, 1460, 910], "label": "Save button" },
  { "id": 3, "bbox": [340, 200, 890, 400], "label": "text editor area" }
]

Why a parser beats referent-style grounding for Horus

Referent-style grounding (OS-Atlas, UGround etc.) takes a screenshot + query → returns one click point.

Parser style takes a screenshot → returns ALL interactive elements exhaustively.

The parser argument: you cannot trust any VLM - including Sonnet or GPT-4o - to reliably locate a 16px checkbox in a dense UI. Offloading exhaustive detection to a specialized model means the reasoning model never has to "see" the screen carefully. It just reasons over a structured list. More reliable division of labor.

SoM (Set-of-Mark) rendering

After OmniParser returns the element list, numbered markers are drawn directly onto the screenshot:

Screenshot → OmniParser → element list → SoM renderer → annotated screenshot

The reasoning model receives BOTH:

  • The annotated screenshot (visual - sees the markers)
  • The element list as text (structural - can reason over labels)

Two aligned representations of the same UI state. Consistently the highest-signal input format across grounding benchmarks.

Grounding interface (swappable)

class GroundingBackend(Protocol):
    def parse(self, screenshot: PIL.Image) -> list[UIElement]:
        ...

@dataclass
class UIElement:
    id: int
    bbox: tuple[int, int, int, int]  # x1, y1, x2, y2
    label: str
    center: tuple[int, int]          # computed from bbox

OmniParser is the default implementation. Any alternative (Florence-2 only, accessibility tree, OS-Atlas) implements the same interface. Horus core never imports OmniParser directly - it calls the interface. This also resolves the AGPL concern: OmniParser runs as a separate sidecar process, Horus talks to it over HTTP. Inter-process communication is not linking.

5. Action DSL

Why a DSL

Natural language action output = unparseable, ambiguous, inconsistent across models.

A grammar constrains the output space so actions either parse or they don't. No ambiguity.

Grammar

action     ::= click(id)
             | type(text)
             | scroll(dir, amount)
             | hotkey(keys)
             | select(id)
             | drag(from_id, to_id)
             | wait(ms)
             | done(result)
             | fail(reason)

id         ::= integer          -- references OmniParser element ID
dir        ::= "up" | "down" | "left" | "right"
amount     ::= integer          -- pixels
keys       ::= string           -- e.g. "cmd+s", "ctrl+z"
text       ::= string
result     ::= string
reason     ::= string

Note: not finalized. will refer to OSWorld action space to see if this language is enough and update throughout development.

How element IDs become clicks

The model never outputs raw coordinates. It outputs element IDs. The execution backend resolves them and performs the action - the agent loop never touches the OS directly.

class ExecutionBackend(Protocol):
    async def click(self, x: int, y: int) -> None: ...
    async def type_text(self, text: str) -> None: ...
    async def hotkey(self, keys: str) -> None: ...
    async def screenshot(self) -> PIL.Image: ...

async def execute_action(
    action: Action,
    elements: list[UIElement],
    backend: ExecutionBackend
) -> None:
    if action.type == "click":
        element = elements[action.id]
        x, y = element.center          # resolved from bounding box
        await backend.click(x, y)      # backend handles OS call
    elif action.type == "type":
        await backend.type_text(action.text)
    elif action.type == "hotkey":
        await backend.hotkey(action.keys)

The backend is swappable behind the interface - the agent loop never changes:

ExecutionBackend
├── PyAutoGUIBackend      - local dev, fast iteration
├── RustBackend           - production Tauri app
├── OSWorldBackend        - benchmark harness
└── VNCBackend            - remote machine control

The model reasons semantically ("click the Save button → id 2"), the agent loop resolves the ID to coordinates, the execution backend handles the OS call. Three clean layers, none of which know about each other.

OSWorld action space alignment

OSWorld expects this format:

{
    "action_type": "click",
    "x": 1420,
    "y": 890
}

The DSL maps to this trivially - element ID → center coordinates → OSWorld format. OSWorld compatibility is free if you build the DSL correctly from day one.

Constrained generation

If the user is running a local model, they can use Outlines or equivalent to enforce the grammar at the token-sampling level through constrained sampling. The model physically cannot produce a malformed action token. No output parsing failures. For API calls, we can validate return schema programmatically and send back an error message.

6. Agent Loop

Core principle

One API call = one think + one action. The loop structure enforces interleaved reasoning without any model-specific beta headers or API features. Works with any OpenRouter model.

Single call structure

SYSTEM = """
You are a computer use agent.
You will receive:
- A screenshot with numbered markers on interactive elements
- A list of those elements with their labels
- The task you are completing
- The history of actions taken so far

Reason about the current state, then output exactly one action on the last line as JSON.
Valid actions: click(id), type(text), scroll(dir, amount), hotkey(keys), done(result), fail(reason)

Example output:
The save dialog is open. Element 4 is the confirm button. I should click it.
{"action": "click", "id": 4}
"""

Each call produces: reasoning (free text) + action (JSON on last line). Parse the JSON off the bottom.

Loop implementation reference

async def agent_loop(task: str) -> str:
    messages = [{"role": "user", "content": task}]

    while True:
        # 1. capture screen
        screenshot = capture_screenshot()

        # 2. perception
        elements = grounding_backend.parse(screenshot)
        annotated = som_renderer.render(screenshot, elements)

        # 3. build context
        messages[-1]["content"] = build_context(
            screenshot=annotated,
            elements=elements,
            task=task,
            history=messages
        )

        # 4. model call - one think + one action
        response = await call_model(messages)
        reasoning, action = parse_response(response.content)

        # 5. append full turn to history
        messages.append({"role": "assistant", "content": response.content})

        # 6. confidence gate
        if not confidence_check(action, elements):
            await recovery_handler(messages)
            continue

        # 7. execute
        if action.type == "done":
            return action.result
        if action.type == "fail":
            raise AgentFailure(action.reason)

        result = execute_action(action, elements)

        # 8. feed result back
        messages.append({
            "role": "user",
            "content": f"Action executed. Result: {result}"
        })

7. Context Management

The problem

Each step adds to history:

  • Reasoning trace (verbose text)
  • Action JSON
  • Tool result
  • Screenshot (extremely token-heavy)

A 30-step task with raw screenshots at every step blows most context windows and costs a lot.

Strategy: four-layer compression

flowchart LR
    A[Raw trajectory] --> B{Step age}
    B -->|Last 3 steps| C[Full screenshot + full reasoning]
    B -->|Steps 4-10| D[Structured diff, what changed only]
    B -->|Steps 11-20| E[Text summary, no screenshot]
    B -->|Steps 20+| F[Single paragraph, high level only]

Write - externalize state to a scratchpad file, not just context

Select - retrieve only relevant history for current step

Compress - summarize old trajectory segments, drop raw screenshots

Isolate - subagents handle their own context, return summaries only

Screenshot compression

Instead of passing full screenshots for old steps, pass structured diffs:

# Instead of: [screenshot at step 8]
# Pass: "Step 8: File dialog closed. Text field now shows '/Users/hari/docs'. Button 'Open' appeared."

OmniParser makes this tractable - diff the element lists between steps, describe what appeared/disappeared/changed. Dramatically cheaper than pixel data.

Note: Great idea. Should be top of mind during implementation.

8. Subagent Architecture

When to decompose

The orchestrator decomposes tasks it classifies as multi-phase. Each subagent gets an isolated context window with only what it needs.

flowchart TD
    A[Orchestrator
    full task context] -->|subtask 1 brief| B[Subagent 1
    isolated context]
    A -->|subtask 2 brief| C[Subagent 2
    isolated context]
    A -->|subtask 3 brief| D[Subagent 3
    isolated context]
    B -->|structured summary| A
    C -->|structured summary| A
    D -->|structured summary| A
    A --> E[Final result]

Note: Look into alternative agent architectures, like slate's thread weaving and episodes Read about ReACT and RLM (Prime Intellect Blog).

What each subagent receives

subagent_context = {
    "global_task": "Book a flight from Delhi to Tokyo for March 28",
    "your_subtask": "Navigate to Google Flights and find available flights",
    "current_screen": annotated_screenshot,
    "elements": element_list,
    # nothing else - no full orchestrator history
}

What it returns

subagent_result = {
    "subtask": "Navigate to Google Flights and find available flights",
    "status": "completed",
    "summary": "Opened Google Flights. Found 3 available flights on March 28. Cheapest is Air India at ₹45,000 departing 06:30.",
    "verified_state": "Flight results page showing 3 options"
}

The orchestrator never sees raw subagent trajectories. Only summaries. Context stays lean.

9. Confidence Gating

Why not multi-rollout (BJudge style)

Multi-rollout (run N agents, pick best) is a benchmark-maximizing technique. In real deployment:

  • Most tasks have side effects (you can't book 5 flights and pick one)
  • N× latency and cost
  • Judge is still just another model

Confidence gate instead

Score confidence at key checkpoints - after navigation, after form interactions, after state-changing actions. Not every step.

def confidence_check(action: Action, elements: list[UIElement], history: list) -> bool:
    # Hard checks
    if action.type == "click" and action.id not in [e.id for e in elements]:
        return False  # element doesn't exist on screen

    # Soft checks - ask model to self-assess
    score = self_assessment_call(action, elements, history)
    return score >= CONFIDENCE_THRESHOLD

On failure: attempt local recovery (go back, retry from last verified state) before full restart. Single rollout, recoverable not restartable.

Note: Think about comprehensively enumerating different types of confident gates and verbose error handling that returns error messages to the orchestrator instead of crashing.

10. OSWorld Integration

How OSWorld works

sequenceDiagram
    participant O as OSWorld Controller
    participant V as VirtualBox VM
    participant H as Horus Agent

    O->>V: Reset to task snapshot
    O->>H: Send task instruction
    loop Agent loop
        H->>V: get_screenshot()
        V-->>H: PIL Image
        H->>H: Perceive + reason + act
        H->>V: execute_action(action_dict)
    end
    O->>V: Evaluate final state
    O-->>H: Score

Minimal integration

from desktop_env.envs.desktop_env import DesktopEnv

env = DesktopEnv(action_space="computer_13")
obs, info = env.reset(task_config=task)

while True:
    screenshot = obs["screenshot"]        # PIL Image - goes straight to OmniParser
    instruction = obs["instruction"]

    action = horus_agent.predict(screenshot, instruction)   # your agent
    obs, reward, done, info = env.step(action)

    if done:
        break

score = env.evaluate()

Horus' agent output maps to OSWorld's 13-action format via the DSL resolver. OSWorld compatibility is a one-day integration, not a week.

Note: This ties into how the backend parses actions into different destinations, as discussed in [[#How element IDs become clicks]].

13. Development Notes

Core loop is observe -> reason -> execute.

  1. The observation layer includes getting information from the desktop environment, be it a VM, remote docker container, or the users laptop.
  2. The reasoning layer is where the bulk of the work is, and all of the agent architecture based optimisations live. We define a standard output DSL, which we can then parse in the execution layer to OS level code implemented using Rust, pyautogui, or anything else.
  3. The execution layer takes the output DSL from the reasoning layer, and parses it into OS code for pyautogui, OSWorld's action space, Rust, or anything else.
